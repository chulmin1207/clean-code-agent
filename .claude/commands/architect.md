$ARGUMENTS 에 지정된 디렉토리의 아키텍처를 분석하세요.

## 전제 조건

- $ARGUMENTS 가 없으면 프로젝트 루트(src/ 또는 .)를 자동 대상으로 선택
- 소스 파일이 없으면 에러 출력 후 종료

## 분석 항목

### 1. import 그래프 구성
- Glob으로 대상 디렉토리의 모든 소스 파일 수집
- Grep으로 각 파일의 import/require/from 문 추출
- 상대 경로를 절대 경로로 해석
- 파일 간 의존성 맵 생성

### 2. 순환 의존성 탐지
- import 그래프에서 사이클 탐지
- 발견된 사이클을 경로로 출력: A → B → C → A
- 해결 방안 제시 (인터페이스 추출, 이벤트 기반 디커플링 등)

### 3. 레이어 위반 탐지
- 디렉토리 이름 기반 레이어 매핑:
  - Layer 1 (진입점): routes, controllers, handlers, pages, app
  - Layer 2 (비즈니스): services, usecases, domain, logic
  - Layer 3 (데이터): repositories, models, entities, db, store
  - Layer 4 (유틸): utils, lib, helpers, common, shared
- 하위 레이어가 상위 레이어를 import하면 위반

### 4. 결합도 측정
- fan-in: 이 파일을 import하는 파일 수
- fan-out: 이 파일이 import하는 파일 수
- fan-out > 10 → God 모듈 의심
- fan-in > 15 → 변경 영향 범위 과대

### 5. God 파일 탐지
- wc -l로 파일별 줄 수 측정
- 300줄 초과 목록 생성
- Grep으로 export 수 카운팅
- 300줄 + export 10개 이상 → God 파일

### 6. 모듈 응집도 평가
- 같은 디렉토리 내 파일들이 서로 얼마나 import하는지 측정
- 디렉토리 내부 의존 비율이 낮으면 응집도 부족 (관련 없는 파일이 같은 폴더에)

## 출력 형식

```
## 아키텍처 분석: {디렉토리}

### 요약
- 소스 파일: {N}개
- 총 의존성 엣지: {N}개
- 평균 fan-out: {N}
- 순환 의존성: {N}개
- 레이어 위반: {N}개
- God 파일: {N}개

### 순환 의존성
| # | 사이클 | 관련 파일 | 해결 방안 |
|---|--------|-----------|-----------|

### 레이어 위반
| 위반 파일:라인 | import 대상 | 위반 유형 | 해결 방안 |
|----------------|------------|-----------|-----------|

### 결합도 경고
| 파일 | fan-in | fan-out | 경고 |
|------|--------|---------|------|

### God 파일
| 파일 | 줄 수 | export 수 | 분리 제안 |
|------|-------|-----------|-----------|

### 아키텍처 점수: {점수}/10
- 순환 의존 0개: 10점, 1개당 -3
- 레이어 위반 0개: 보너스 없음, 1개당 -2
- God 파일 0개: 보너스 없음, 1개당 -1

### 개선 로드맵
1. (가장 치명적인 순환부터)
```
